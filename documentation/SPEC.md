# ローカルAIチャット履歴ビューア 仕様書

既存のデータは絶対に破壊してはいけない。

## システム概要

関連設計資料: [documentation/DATA_MODEL.md](documentation/DATA_MODEL.md)

Windows ローカルに保存されている GitHub Copilot Chat のチャット履歴を抽出し、一覧表示・閲覧・Markdown 出力を行うデスクトップアプリケーション。

対象は GitHub Copilot Chat のみとし、VS Code の他拡張機能や一般的な AI チャット保存形式は対象外とする。標準的なインストール環境では「起動即スキャン」を実現しつつ、特殊な環境にも対応できる柔軟性を備える。

## 動作環境

* OS: Windows 10/11
* 言語: Python 3.12
* UI: Tkinter

## 対象拡張・識別方針

* 対象拡張は GitHub Copilot Chat とする。
* 拡張識別子は `github.copilot-chat` を前提とする。
* 現行形式の主要な保存先は `workspaceStorage\{workspaceHash}\GitHub.copilot-chat\` 配下とする。
* `debug-logs\` は診断用ログとして扱い、通常の会話本文表示対象から除外する。
* `memory-tool\` は会話本文ではないため、通常の会話本文表示対象から除外する。

## パス解決・探索ロジック

アプリ起動時、以下の優先順位で解析対象ディレクトリを決定する。

### デフォルトパスの自動検知

1. 環境変数 `%APPDATA%\Code\User` をデフォルトの探索起点とする。
2. 起動時に上記ディレクトリ配下の `workspaceStorage\*\GitHub.copilot-chat\transcripts\` を優先探索する。
3. 上記が見つからない場合、補助メタデータとして `globalStorage\github.copilot-chat\` 配下、および必要に応じて `.vscdb` を探索する。
4. 現行形式または後方互換形式の有効なデータが見つかった場合、ユーザー操作を待たず自動的にスキャンを開始し、履歴一覧を表示する。

### 任意パスの手動指定 (フォールバック)

以下の状況において、ユーザーが任意のディレクトリを選択できる UI を提供する。

* デフォルトパスが存在しない場合（VS Code Insiders 利用時、ポータブル版利用時など）。
* ユーザーが特定のバックアップフォルダや特定のワークスペースのみを指定したい場合。
* UI 上の「フォルダを選択」ボタンから `%APPDATA%` 外の場所も指定可能とする。

## 入力データ仕様

### 優先順位

1. 主記録: `workspaceStorage\{workspaceHash}\GitHub.copilot-chat\transcripts\{sessionId}.jsonl`
2. 補助メタデータ: `globalStorage\github.copilot-chat\copilotcli\` 配下の JSON ファイル
3. 後方互換フォールバック: `.vscdb` (SQLite3)

### 主記録として扱うファイル

1. `transcripts\{sessionId}.jsonl`
    * GitHub Copilot Chat の会話本文を保持する現行形式として扱う。
    * 1 行目に `type = "session.start"` を持ち、`data.sessionId` がファイル名の `{sessionId}` と一致することを有効条件とする。
    * 各行は JSON オブジェクトであり、少なくとも `type`、`data`、`timestamp`、`id` を持つことを期待する。

### 補助メタデータとして扱うファイル

1. `globalStorage\github.copilot-chat\copilotcli\copilotcli.session.metadata.json`
    * セッション本文の代替ではなく、タイトル、ワークツリー、ワークスペース情報などの補助情報として扱う。
2. `globalStorage\github.copilot-chat\copilotcli\{sessionId}\vscode.metadata.json`
    * セッション単位の補助メタデータとして扱う。
3. `globalStorage\github.copilot-chat\copilotcli\{sessionId}\vscode.requests.metadata.json`
    * リクエスト対応情報の補助メタデータとして扱う。

### 除外対象

* `workspaceStorage\{workspaceHash}\GitHub.copilot-chat\debug-logs\**`
  * デバッグおよび診断用途であり、通常の会話一覧生成には使用しない。
* `workspaceStorage\{workspaceHash}\GitHub.copilot-chat\memory-tool\**`
  * メモリ機能の内部保存であり、通常の会話一覧生成には使用しない。

### SQLite フォールバック

* `.vscdb` は後方互換のための補助入力としてのみ扱う。
* SQLite ファイルが VS Code により使用中の場合でも、可能な限り読み取り専用で開くことを優先する。
* 読み取り方法は SQLite URI または同等手段による read-only 接続を前提とし、書き込みや更新は一切行わない。
* 読み取り専用でのオープンに失敗した場合は当該ファイルをスキップし、アプリ全体は継続動作する。
* `.vscdb` 内では `ItemTable` を対象とし、GitHub Copilot Chat に明確に紐づくキー空間または既知スキーマに一致するデータのみを採用する。

## データ統合・重複解決

### セッション識別子

* セッションの主キーは `sessionId` とする。
* 現行形式では transcript ファイル名、および `session.start.data.sessionId` を正とする。

### 重複時の優先順位

同一 `sessionId` の情報が複数ソースに存在する場合、以下の優先順位で統合する。

1. `transcripts\{sessionId}.jsonl` の本文とタイムスタンプを正とする。
2. `copilotcli.session.metadata.json` およびセッション単位メタデータは、本文を上書きせず補助情報としてマージする。
3. `.vscdb` は transcript または JSON メタデータに存在しない項目の補完にのみ使用する。

### 競合解決ルール

* 会話本文は常に transcript JSONL を優先する。
* `debug-logs` の内容で本文を再構成しない。
* タイトルや補助情報が競合する場合は、現行 JSON ファイル由来の値を優先する。
* セッション本文が存在しないレガシーデータのみの場合に限り、`.vscdb` 由来セッションを一覧対象に含める。

## 機能要件

### 非対象

* GitHub Copilot Chat 以外の拡張機能の履歴解析は対象外とする。
* クラウド同期、外部API連携、オンライン共有機能は対象外とする。
* 元データの修復、更新、削除、移行は対象外とする。
* `debug-logs` からの本文再構成や、診断ログの可視化は対象外とする。

### 自動スキャンとステータス表示

* 起動時、自動探索の結果を UI 上に表示すること。
* 例: 「Copilot Chat セッションをスキャン中...」「有効なセッションを 12 件検出」など。
* 有効なチャットデータが見つからない場合は、手動選択を促すメッセージを表示すること。
* 読み取り専用で開けなかった `.vscdb` がある場合は、警告表示のみ行い処理継続すること。

### セッション一覧とソート

* 左ペインにセッション一覧を表示する。
* 既定のソート基準は「セッション内の最新タイムスタンプ」とする。
* 追加のソート基準として「セッション内の最古タイムスタンプ」を選択可能とする。
* 各ソート基準に対して、降順 / 昇順を切り替え可能とする。
* ファイルの更新日時は一覧ソートの正規基準として使用しない。

### 詳細閲覧

* 選択したセッションの内容を右ペインに表示する。
* ユーザーと AI の発言を色分け、またはラベルで区別して表示する。
* コードブロック等は等幅フォントで表示する。
* JSONL のイベント列から、通常の会話閲覧に不要な内部イベントは必要に応じて非表示または整形して表示する。

### Markdown エクスポート

* 閲覧中の単一セッションを `.md` 形式で保存できることを必須要件とする。
* ファイル名はセッション日時をベースに自動生成し、保存ダイアログで変更可能とする。
* 可能であれば複数選択エクスポートに対応する。
* 可能であれば一覧上の全選択一括エクスポートに対応する。
* 複数エクスポート時は、1 セッション 1 ファイルを基本とする。

## パーサ実装方針

* JSONL では既知のイベント型を優先的に解釈し、未知のイベント型は無視または内部イベントとして折りたたむ。
* 表示対象の本文は、少なくともユーザー発話とアシスタント発話を復元できることを優先する。
* `session.start` は一覧生成と検証に使用し、会話本文としては表示しない。
* タイムスタンプ欠損時は、そのイベントを並び順の主判定に使わず、他イベントからセッション時刻を推定する。
* 1 ファイルの一部行が不正JSONでも、読める行だけで可能な限りセッションを復元する。

## 受け入れ条件

* `%APPDATA%\Code\User` 配下に現行形式の transcript が存在する環境で、起動後に自動でセッション一覧が表示されること。
* 一覧から任意のセッションを選択し、ユーザーとアシスタントの発話を判別して閲覧できること。
* 一覧を「最新タイムスタンプ基準」と「最古タイムスタンプ基準」で昇順・降順に並び替えできること。
* 単一セッションを Markdown として保存できること。
* 一部ファイルが破損していても、アプリ全体が異常終了せず、読めるセッションだけ表示できること。
* `.vscdb` がロック中であっても、読み取り専用で扱うか、読めない場合はスキップして継続できること。

## 非機能要件

* ゼロ・コンフィギュレーション: インストール済みの VS Code に対しては、設定不要で即座にデータへアクセスできること。
* プライバシーの担保: 外部通信を一切行わず、データの読込・表示・エクスポートをローカル環境内で完結させること。
* 安全性: 元データに対して書き込みを行わず、特に `.vscdb` は必ず読み取り専用で扱うこと。
* 耐障害性: 一部ファイルの破損、スキーマ差異、ロック失敗があっても、読めるセッションだけで継続動作すること。
* メンテナンス性: GitHub Copilot Chat の現行保存形式を優先しつつ、後方互換のフォールバックを分離した設計とすることで、保存形式変更時の影響範囲を限定すること。

既存のデータは絶対に破壊してはいけない。
